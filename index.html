Core Concepts
📦 Repository (Repo)▼

A repository is your project folder that Git tracks. It contains your files plus a hidden .git folder that stores all history.

# Create a new repo in current folder
git init

# Clone an existing repo from GitHub
git clone https://github.com/user/repo.git
💾 Commit▼

A commit is a snapshot of your project at a specific moment. Each commit has a unique ID (SHA), an author, date, and message.

git add . # Stage all changes
git commit -m "Add login feature"

[main a3f2c1] Add login feature
2 files changed, 45 insertions(+)

Think of commits like save points in a video game — you can always reload from any save point.

🎯 HEAD▼

HEAD is a pointer to the commit you're currently at. It's like a cursor — it moves as you create commits or switch branches.

🔄 Push & Pull▼

Push uploads your commits to GitHub. Pull downloads teammates' commits to your machine.

git push origin main # Upload to GitHub
git pull origin main # Download from GitHub
📋 Common Commands▼
# Setup
git config --global user.name "You"
git config --global user.email "you@email.com"

# Daily workflow
git status # What changed?
git log --oneline # Commit history
git diff # See exact changes
git stash # Save temp changes

📈 Commit History

Each commit points back to its parent — forming a chain of history.

Branching

Work in Parallel

🌿 What is a Branch?

A branch is a parallel version of your code. You can create a new branch to work on a feature without affecting the main codebase. It's like a parallel universe for your code.

Branch Commands
# Create and switch to new branch
git checkout -b feature/login

# Or with newer syntax
git switch -c feature/login

# List all branches
git branch

# Merge feature into main
git checkout main
git merge feature/login

# Delete merged branch
git branch -d feature/login
⚠️ Merge Conflicts▼

A merge conflict happens when two branches changed the same line differently. Git marks the conflict in the file:

<<<<<<< HEAD
const greeting = "Hello";
=======
const greeting = "Hi there";
>>>>>>> feature/login

You must manually pick which version to keep, then git add and git commit to resolve it.

🔄 Git Flow Strategy

A popular branching strategy used in teams:

main Production-ready code only
develop Integration branch for features
feature/* New features, one branch per task
hotfix/* Emergency fixes for production
release/* Preparing a new version release
GitHub

Git in the Cloud

🐙 What is GitHub?

GitHub is a cloud platform that hosts your Git repositories online. It adds collaboration features on top of Git: Pull Requests, Issues, Actions, Code Review, and more. It's where the world's code lives.

Git vs GitHub

⚡ Git

  • Works offline
  • Local tool
  • Tracks history
  • Free & open source
  • Command line
  • No account needed

☁️ GitHub

  • Needs internet
  • Cloud platform
  • Hosts repos
  • Free tier + paid
  • Web UI + more
  • Account required

🔗 Remote Repository

A remote is a version of your repo hosted on the internet. "origin" is the default name for your GitHub remote.

Fork vs Clone
📋 Clone
Downloads a copy of a repo to your computer. You still push to the original repo if you have access. Used when you have permission to contribute.
git clone https://github.com/org/repo.git
🍴 Fork
Creates a copy on GitHub under your account. You push to your fork, then submit a Pull Request to the original. Used for open source contributions.
Pull Request Flow

🔀 What is a Pull Request?

A PR is a formal request to merge your changes into the main codebase. It allows team review before code is merged.

1
Fork or branch the repo, make your changes locally
2
Push your branch to GitHub: git push origin feature/x
3
On GitHub, click "New Pull Request" and describe your changes
4
Team reviews the code, leaves comments, requests changes
5
After approval, maintainer clicks "Merge Pull Request" ✅
